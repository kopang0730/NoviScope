# NoviScope Collaboration Guide

这份文档用于约束小组成员和 AI 工具共同参与 NoviScope 开发时的协作方式。目标不是增加流程负担，而是避免项目变成不可维护、不可审查、不可复现的代码堆。

## 基本原则

- 先写清楚问题，再写代码。
- 小步提交，小 PR 合并。
- 每个行为变化都应该有测试或明确的人工验证记录。
- AI 可以辅助实现，但不能替代人对设计、风险和结果负责。
- 涉及科研结论、实验结果、文献引用和数据安全的变更必须保留证据链。

## Issue 使用规则

每个非平凡任务都应该先开 issue。issue 至少说明：

- 背景和目标
- 预期行为
- 不做什么
- 验证方式
- 相关风险

适合拆 issue 的粒度：

- 一个 API 功能
- 一个 agent 能力
- 一个数据模型变更
- 一个实验 runner 能力
- 一份文档或模板

不建议把多个大模块放到一个 issue 里，例如“实现完整科研系统”。

## Pull Request 规则

PR 应该回答：

- 改了什么
- 为什么这样改
- 有哪些风险
- 怎么验证
- 是否影响已有 API、数据库或 agent 权限

PR 不应该包含：

- 未说明来源的大段 AI 生成代码
- 未验证的论文结论
- API key、私有数据、实验日志、checkpoint
- 与任务无关的格式化或重构

## AI 生成代码规范

允许使用 AI 写代码，但必须满足：

- 代码经过人类阅读和测试。
- PR 描述中说明哪些部分由 AI 辅助。
- 不接受“AI 说可以”的验证方式。
- 不接受无法解释的数据模型、状态机、权限控制或安全逻辑。
- 不把未验证的文献观点、实验结果写进 README、论文或报告。

## 代码审查重点

审查优先级：

1. 是否会导致数据泄露或密钥泄露。
2. 是否破坏 quest/stage 状态机。
3. 是否让 agent 获得过宽权限。
4. 是否把未验证结论当作事实。
5. 是否缺少测试。
6. 是否引入难以维护的大文件或重复逻辑。

## Definition of Done

一个任务完成至少需要：

- 功能实现或文档更新已经提交。
- `ruff check .` 通过。
- `pytest` 通过，或说明失败原因。
- 对外行为经过 API、CLI 或文档检查。
- GitHub issue 或 PR 中有验证记录。

## 研究可信度约束

NoviScope 的最终目标是辅助科研，不是批量生成看似完整的内容。因此：

- 文献结论必须能追溯到来源。
- 实验结果必须能追溯到代码、参数、数据和日志。
- 论文结论必须由 evidence auditor 认可后才能进入正式草稿。
- 需求真实性验证未通过前，不应进入高成本实验阶段。
